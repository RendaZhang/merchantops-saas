## Overview

MerchantOps SaaS now sits on a completed Week 1-9 working-tree baseline with Week 10 Delivery Hardening and Portfolio Packaging active. The public surface covers tenant-scoped user management, one real tenant-scoped feature-flag management API, ticket workflow, audit/approval, import jobs, read-only ticket AI summary, triage, internal reply-draft, entity-scoped interaction-history endpoints, one tenant-scoped AI usage-summary endpoint, four read-only import AI endpoints, and two completed Week 8 human-reviewed execution bridges: approval-bounded import selective replay and approval-bounded ticket comment creation from AI reply-draft output. Week 8 governance hardening now also ships as the shared approval baseline: pending proposals are deduplicated on executable payload semantics, concurrent duplicate creates collapse to one pending row, and resolved requests release their pending key for future proposals. Week 9 is now complete as the executable prompt-inventory and eval-comparator baseline together with the tenant-scoped usage/cost governance read surface and prompt-version aggregate visibility on that same tenant summary. Week 10 Slice A is now complete as persisted rollout control for the six AI generation endpoints and the two Week 8 workflow bridges.

## Current Repository Baseline

### Public workflow surface

- Auth and context: `POST /api/v1/auth/login`, `GET /api/v1/user/me`, and `GET /api/v1/context`.
- User management: tenant-scoped list/detail/create/update/status/role-assignment plus role lookup and disable-request initiation.
- Feature flags: tenant-scoped `GET /api/v1/feature-flags` and `PUT /api/v1/feature-flags/{key}` under `FEATURE_FLAG_MANAGE`, exposing one fixed eight-flag set per tenant for AI generation and workflow-bridge rollout control, with `PUT` requiring a non-null boolean `enabled` value, returning the final persisted row, and skipping duplicate audit rows when a concurrent write already reached the requested state.
- Ticket workflow: tenant-scoped list/detail/create/assignee/status/comment flow with queue filters, reopen support, and a separate ticket comment proposal endpoint for human-reviewed reply-draft execution.
- AI-assisted ticket read path: `GET /api/v1/tickets/{id}/ai-interactions` for narrowed interaction-history visibility plus operator-visible runtime usage/cost metadata, `POST /api/v1/tickets/{id}/ai-summary` for suggestion-only summaries, `POST /api/v1/tickets/{id}/ai-triage` for suggestion-only classification and priority guidance, and `POST /api/v1/tickets/{id}/ai-reply-draft` for internal comment-style reply drafts from ticket detail context.
- AI governance read path: `GET /api/v1/ai-interactions/usage-summary` for tenant-scoped aggregate runtime metadata across stored ticket and import AI rows, with inclusive `from` / `to`, optional exact-match `entityType` / `interactionType` / `status` filters, aggregate token/cost totals, stable `byInteractionType` / `byStatus` / `byPromptVersion` ordering, and no per-request cross-entity detail listing.
- AI-assisted import read path: `GET /api/v1/import-jobs/{id}/ai-interactions` for narrowed stored import AI interaction history with operator-visible runtime usage/cost metadata when present; `POST /api/v1/import-jobs/{id}/ai-error-summary` for a suggestion-only summary built from current-tenant import detail, `errorCodeCounts`, and the first sanitized failed-row window without forwarding raw CSV payload values to the provider; `POST /api/v1/import-jobs/{id}/ai-mapping-suggestion` for a suggestion-only canonical-field mapping proposal built from sanitized header/global parse signal plus the same bounded row-summary context; and `POST /api/v1/import-jobs/{id}/ai-fix-recommendation` for a suggestion-only fix recommendation built from grounded row-level `errorCode` groups without returning replacement values.
- Governance: entity-scoped `GET /api/v1/audit-events` plus an action-aware approval request queue/detail/approve/reject flow for `USER_STATUS_DISABLE`, `IMPORT_JOB_SELECTIVE_REPLAY`, and `TICKET_COMMENT_CREATE`, with shared pending-request-key deduplication on executable payload semantics and key release after approval resolution.
- Import jobs: tenant-scoped create/list/detail/ai-interactions/ai-error-summary/ai-mapping-suggestion/ai-fix-recommendation/replay/replay-file/selective-replay/selective-replay-proposal/edited-replay/error-page flow with `USER_CSV` processing, filtered queue reads, quoted CSV record parsing, `errorCodeCounts`, row-level item errors, replay-derived job lineage, scope-only replay audit metadata, and the current Week 5 plus Week 7 plus Week 8 reporting/guidance surface.

## Current Public AI Baseline

- The current public AI slices are `GET /api/v1/tickets/{id}/ai-interactions`, `POST /api/v1/tickets/{id}/ai-summary`, `POST /api/v1/tickets/{id}/ai-triage`, `POST /api/v1/tickets/{id}/ai-reply-draft`, `GET /api/v1/ai-interactions/usage-summary`, `GET /api/v1/import-jobs/{id}/ai-interactions`, `POST /api/v1/import-jobs/{id}/ai-error-summary`, `POST /api/v1/import-jobs/{id}/ai-mapping-suggestion`, and `POST /api/v1/import-jobs/{id}/ai-fix-recommendation`.
- All nine slices are read-only. The six `POST` endpoints remain suggestion-only: they do not change ticket status, write comments, mutate import jobs, mutate import error rows, or trigger approvals or replay flows, and the three `GET` endpoints only expose narrowed stored metadata or aggregate runtime metadata.
- The six generation endpoints now require both config-level `merchantops.ai.enabled=true` and their matching persisted feature flag, while `GET /api/v1/tickets/{id}/ai-interactions`, `GET /api/v1/import-jobs/{id}/ai-interactions`, and `GET /api/v1/ai-interactions/usage-summary` remain available when generation is gated off.
- Reply-draft semantics are explicitly internal and comment-style: the endpoint returns a structured draft plus `draftText`, but still does not create a comment or send an external message.
- The current AI context is limited to the target current-tenant ticket's core fields plus the most recent comments and workflow logs, with prompt-time truncation and explicit `earlier ... omitted` markers when older history exists.
- The import AI context is limited to current-tenant import detail, `errorCodeCounts`, sanitized header/global parse-error signal, grounded row-level `errorCode` groups, and the first 20 failed rows after local structural sanitization; raw `itemErrors.rawPayload` values, including `USER_CSV.password`, are not forwarded to the provider, and fix recommendation additionally rejects sensitive echoed output before returning a response.
- The interaction-history surfaces return both successful and controlled-failure records with exact-match `interactionType` and `status` filters plus stable `createdAt DESC, id DESC` ordering.
- The interaction-history surfaces now expose ticket-scoped and import-scoped runtime usage/cost metadata when present, return those fields as `null` when unavailable, and still do not expose raw prompt text or raw provider payload.
- The tenant-scoped usage-summary surface aggregates current-tenant `ai_interaction_record` rows across `TICKET` and `IMPORT_JOB`, supports optional inclusive `from` / `to` plus exact-match trimmed `entityType` / `interactionType` / `status` filters, returns totals plus `byInteractionType` / `byStatus` / `byPromptVersion` breakdowns, and intentionally excludes per-request fields such as `requestId`, `outputSummary`, and `modelId` while exposing `promptVersion` only as aggregate prompt-version buckets.
- The current provider ownership model is instance-level configuration rather than tenant BYOK.
- The current Week 9 governance baseline now includes an executable six-workflow inventory, one shared comparator pass for golden, failure, and policy datasets, and one tenant-scoped usage-summary read surface over stored runtime metadata with prompt-version aggregate visibility; broader reporting still comes after that narrow baseline rather than before it.
- The completed Week 6 ticket AI baseline and the completed Week 7 import AI read baseline stay stable while the completed Week 8 workflow now adds two separate approval-backed execution bridges: import selective replay proposals and ticket comment proposals. The nine public AI endpoints themselves still do not write back or self-execute, and the two workflow bridges are now separately controllable through persisted workflow flags.

## Current Limitations

- Import jobs currently support one business import type only: `USER_CSV`.
- The `USER_CSV` schema is fixed to `username,displayName,email,password,roleCodes`.
- Approval flow currently covers three action types only: `USER_STATUS_DISABLE`, `IMPORT_JOB_SELECTIVE_REPLAY`, and `TICKET_COMMENT_CREATE`, but pending uniqueness now spans all three rather than only disable.
- Audit reads are still minimal and entity-scoped by `entityType + entityId`, and the new tenant AI usage-summary endpoint is still only a narrow aggregate governance read rather than a general reporting system.
- `UserCommandService#updatePassword` remains a placeholder business error, not a completed write flow.
- There is no refresh-token flow, logout flow, or token revocation flow yet.
- Public AI now includes ticket interaction history, one tenant usage-summary surface, ticket summary/triage/internal reply draft, and import AI interaction history plus error summary plus mapping suggestion plus fix recommendation; the human-reviewed write bridges remain separate workflow endpoints rather than direct AI endpoint mutations.
- The AI read surfaces expose runtime usage/cost metadata only through entity-scoped interaction history and the tenant-scoped usage summary; they are not tenant billing, ledger, or invoice surfaces.
- AI runtime still uses instance-level provider configuration only; there is no tenant BYOK, streaming, tool calling, model routing, or automatic write-back loop.
- Feature flags are intentionally tenant-scoped and fixed-key only in this slice; there is no cross-tenant admin surface, percentage rollout, multi-environment rollout policy, or generic flag platform yet.
- Ticket enrichments such as priority, SLA, attachments, and notifications remain post-Week-3 follow-up work.
- Deployment-ready manifests, production secret-management guidance, and performance artifacts are still pending.
- There is no tenant admin UI or frontend yet.
